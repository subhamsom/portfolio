The Story

My first app. Built from zero.

A few days before building this, I didn't know what Node.js was. I'd never used a terminal for anything beyond the occasional copy-paste command I found online. The idea of building and deploying a web app felt like something reserved for people who'd spent years studying computer science.

Then I discovered vibe coding — the idea that you can build real software by describing what you want in plain English and letting AI do the heavy lifting. I decided to test it. I followed a tutorial, adapted it to my own idea, and just… started.

Two days later, Dream Catcher was live on the internet. A real app. With a real database. Real AI. Real authentication. Anyone in the world could sign up and use it.

I hit walls constantly — files not being created, databases not connecting, deployment errors I didn't understand. But I worked through every single one. That process taught me more than any course ever could have.

The Journey

How it came together

Step 01
Documentation first
Used VibeCodeDocs to generate a full Product Requirements Document, user flow diagrams, and styling guidelines before writing a single line of code.
Step 02
Claude Desktop built the skeleton
Fed the documentation to Claude Desktop with filesystem access. It generated 31 files — pages, API routes, components, database schema, and configuration — in one session.
Step 03
Connected the services
Set up Supabase (database + Google OAuth), configured Google Cloud Console, added Gemini AI API key, and wired everything together through environment variables.
Step 04
Debugging with Cursor
Fixed multiple issues — click-blocking CSS overlays, foreign key constraint errors, TypeScript compilation failures — by describing problems in plain English to Cursor's AI.
Step 05
Deployed to Vercel
Pushed to GitHub, imported into Vercel, configured production environment variables, and updated OAuth redirect URIs. Live in under an hour.

Reflection

What I learned

I learned that not understanding every line of code doesn't mean you can't build something real. Knowing how to describe a problem clearly, read an error message, and ask the right questions turned out to be the most valuable skills of all.

Before this project, terms like "foreign key constraint", "Row Level Security", "OAuth callback", and "port conflict" meant nothing to me. Now they're part of how I think. I also learned that shipping imperfect things is better than waiting until everything is perfect — the app is live, it works, and it will keep getting better.
